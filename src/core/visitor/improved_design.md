# Visitor模块改进设计方案

## 当前问题分析

### 1. 冗余的抽象层
- `VisitorInstance` 和 `VisitorCore` 提供相同功能
- `GenericVisitorInstance` 增加了不必要的包装
- 多个配置管理方式重复

### 2. 复杂的工厂系统
- 全局注册表可能成为性能瓶颈
- 过度泛化的工厂实现
- 构建器模式与配置构建器重复

### 3. 状态管理问题
- 静态可变状态安全隐患
- 上下文和状态管理分散
- 缺乏统一的生命周期管理

## 改进方案

### 1. 简化核心架构
```
ValueVisitor (原始接口)
    ↓
VisitorCore (统一基础设施)
    ↓
具体访问者实现
```

### 2. 统一配置和上下文
- 合并 `VisitorContext` 和配置管理
- 简化状态管理接口
- 移除不必要的抽象层

### 3. 优化工厂系统
- 移除全局注册表
- 简化工厂接口
- 使用组合而非继承

### 4. 改进错误处理
- 统一错误类型
- 简化错误传播
- 更好的错误上下文

## 具体改进措施

### 1. 移除冗余trait
- 删除 `VisitorInstance` trait
- 将其功能合并到 `VisitorCore`
- 简化 `GenericVisitorInstance`

### 2. 重构工厂系统
- 移除全局注册表
- 简化工厂接口
- 使用更直接的创建方式

### 3. 统一配置管理
- 合并配置构建器
- 简化配置接口
- 减少重复代码

### 4. 改进状态管理
- 移除静态可变状态
- 简化状态接口
- 更好的生命周期管理

## 实施计划

1. **第一阶段**: 移除冗余trait和抽象层
2. **第二阶段**: 重构工厂系统
3. **第三阶段**: 统一配置管理
4. **第四阶段**: 改进状态管理
5. **第五阶段**: 优化错误处理

## 预期收益

- **代码简化**: 减少30-40%的代码量
- **性能提升**: 减少不必要的抽象层开销
- **维护性**: 更清晰的架构和更少的重复代码
- **可扩展性**: 更容易添加新的访问者类型
