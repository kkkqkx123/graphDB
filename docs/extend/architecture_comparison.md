# 全文检索架构方案对比分析

## 概述

本文档对比分析全文检索功能的两种架构方案：
1. **内嵌式架构** - 直接集成到 GraphDB 核心
2. **gRPC 服务架构** - 作为独立服务通过 gRPC 调用

---

## 方案一：内嵌式架构

### 架构图

```
┌─────────────────────────────────────────────────────────────┐
│                      GraphDB 进程                            │
│                                                              │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────────┐ │
│  │   HTTP API  │────│ GraphService│────│ Query Pipeline  │ │
│  └─────────────┘    └─────────────┘    └─────────────────┘ │
│                               │                              │
│                               ▼                              │
│  ┌────────────────────────────────────────────────────────┐ │
│  │              Storage Layer (Redb)                       │ │
│  │  ┌─────────────┐  ┌─────────────┐  ┌─────────────────┐ │ │
│  │  │VertexStorage│  │ EdgeStorage │  │ FulltextIndex   │ │ │
│  │  └─────────────┘  └─────────────┘  │ (Tantivy)       │ │ │
│  │                                    └─────────────────┘ │ │
│  └────────────────────────────────────────────────────────┘ │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 优点

#### 1. 性能优势
- **低延迟**：无网络开销，函数调用级别交互
- **高吞吐**：内存共享，数据拷贝少
- **事务一致性**：与存储层共享事务上下文

#### 2. 架构简洁
- **单一进程**：部署简单，运维成本低
- **统一配置**：一个配置文件管理所有功能
- **错误处理统一**：统一的错误码和日志

#### 3. 功能集成度高
- **事务支持**：全文索引更新参与数据库事务
- **权限控制**：复用现有权限系统
- **监控统一**：统一的指标收集和监控

### 缺点

#### 1. 耦合度高
- **依赖冲突**：Tantivy 依赖可能与现有依赖冲突
- **版本绑定**：全文检索库升级影响整个系统
- **编译时间**：增加编译时间和二进制体积

#### 2. 资源竞争
- **内存占用**：Tantivy 索引缓存与数据库缓存竞争
- **CPU 使用**：索引构建可能影响查询性能
- **磁盘 IO**：索引文件与数据文件竞争 IO

#### 3. 扩展性受限
- **水平扩展困难**：无法独立扩展全文检索能力
- **多租户隔离**：难以实现物理隔离

## 方案二：gRPC 服务架构

### 架构图

```
┌─────────────────────────────────────────────────────────────┐
│                      GraphDB 进程                            │
│                                                              │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────────┐ │
│  │   HTTP API  │────│ GraphService│────│ Query Pipeline  │ │
│  └─────────────┘    └─────────────┘    └─────────────────┘ │
│                               │                              │
│                               ▼                              │
│  ┌────────────────────────────────────────────────────────┐ │
│  │              Storage Layer (Redb)                       │ │
│  │  ┌─────────────┐  ┌─────────────┐  ┌─────────────────┐ │ │
│  │  │VertexStorage│  │ EdgeStorage │  │ gRPC Client     │ │ │
│  │  └─────────────┘  └─────────────┘  └─────────────────┘ │ │
│  └────────────────────────────────────────────────────────┘ │
│                               │                              │
│                               │ gRPC                         │
│                               ▼                              │
└─────────────────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────────────────┐
│              Fulltext Service 进程 (独立)                    │
│                                                              │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────────┐ │
│  │ gRPC Server │────│ IndexManager│────│ Tantivy Index   │ │
│  └─────────────┘    └─────────────┘    └─────────────────┘ │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 优点

#### 1. 解耦与独立性
- **依赖隔离**：全文检索依赖不影响主系统
- **独立升级**：可以独立升级 Tantivy 版本
- **技术栈自由**：服务可以使用不同技术栈

#### 2. 资源隔离
- **独立进程**：故障不影响主数据库
- **资源限制**：可独立限制 CPU/内存使用
- **独立存储**：索引文件与数据文件分离

#### 3. 扩展性
- **水平扩展**：可部署多个全文检索实例
- **负载均衡**：支持请求分发
- **多租户**：物理隔离不同租户数据

### 缺点

#### 1. 性能开销
- **网络延迟**：每次调用增加网络往返
- **序列化开销**：数据需要序列化/反序列化
- **连接管理**：需要维护连接池

#### 2. 复杂性增加
- **部署复杂**：需要部署和管理多个服务
- **服务发现**：需要服务注册与发现机制
- **分布式事务**：跨服务事务处理复杂

#### 3. 一致性问题
- **数据同步**：需要处理主从同步
- **延迟问题**：索引更新有延迟
- **故障恢复**：需要处理服务不可用情况

## 详细对比

### 性能对比

| 指标 | 内嵌式 | gRPC 服务 | 差异 |
|------|--------|-----------|------|
| 单次查询延迟 | < 1ms | 2-5ms | 内嵌式快 3-5 倍 |
| 批量索引速度 | 10,000 doc/s | 5,000 doc/s | 内嵌式快 2 倍 |
| 内存占用 | 共享 | 独立 | gRPC 多 20-30% |
| 并发能力 | 受限于单进程 | 可水平扩展 | gRPC 更优 |

### 运维对比

| 指标 | 内嵌式 | gRPC 服务 | 建议 |
|------|--------|-----------|------|
| 部署复杂度 | 低 | 高 | 内嵌式胜 |
| 监控难度 | 低 | 高 | 内嵌式胜 |
| 故障排查 | 简单 | 复杂 | 内嵌式胜 |
| 版本升级 | 风险高 | 风险低 | gRPC 胜 |
| 资源隔离 | 差 | 好 | gRPC 胜 |

### 功能对比

| 功能 | 内嵌式 | gRPC 服务 | 说明 |
|------|--------|-----------|------|
| 事务支持 | 原生支持 | 需分布式事务 | 内嵌式胜 |
| 实时索引 | 实时 | 近实时（有延迟） | 内嵌式胜 |
| 权限控制 | 复用现有 | 需额外实现 | 内嵌式胜 |
| 多租户隔离 | 逻辑隔离 | 物理隔离 | gRPC 胜 |
| 独立扩展 | 不支持 | 支持 | gRPC 胜 |

## 决策矩阵

### 选择内嵌式架构的场景

✅ **推荐内嵌式的情况**：
1. **单机部署**：个人使用、小型应用
2. **低延迟要求**：实时搜索场景
3. **简单运维**：希望单一二进制文件
4. **事务一致性**：需要强一致性保证
5. **资源受限**：服务器资源有限

### 选择 gRPC 服务架构的场景

✅ **推荐 gRPC 的情况**：
1. **大规模部署**：企业级应用、多租户
2. **高可用要求**：需要服务独立故障恢复
3. **团队分工**：不同团队维护不同服务
4. **技术多样性**：希望使用不同技术栈
5. **独立扩展**：全文检索需要独立扩容

## 混合方案建议

考虑到 GraphDB 的定位（轻量级单节点图数据库），建议采用**内嵌式为主，gRPC 为扩展**的混合架构：

### 核心架构（内嵌式）

```rust
// 默认使用内嵌式
pub enum FulltextBackend {
    Embedded(EmbeddedFulltextProvider),  // 默认
    External(GrpcFulltextClient),        // 可选
}
```

### 配置切换

```toml
# config.toml
[fulltext]
backend = "embedded"  # 或 "grpc"

# gRPC 配置（当 backend = "grpc" 时生效）
[fulltext.grpc]
endpoint = "http://localhost:50051"
timeout_ms = 5000
```

### 优势

1. **默认简单**：单机用户使用内嵌式，零配置
2. **灵活扩展**：需要时可切换到 gRPC
3. **向后兼容**：现有部署不受影响
4. **渐进式演进**：先实现内嵌式，后续添加 gRPC 支持

## 推荐方案

### 针对 GraphDB 的推荐

**首选方案：内嵌式架构**

理由：
1. **项目定位**：GraphDB 是轻量级单节点数据库，内嵌式更符合定位
2. **用户体验**：单一二进制文件，部署简单
3. **性能优先**：图数据库查询通常需要低延迟
4. **资源效率**：单机场景资源共享更高效

**实施建议**：

```
Phase 1: 实现内嵌式全文检索（MVP）
├── Tantivy 集成
├── 基础搜索功能
└── SQL 语法扩展

Phase 2: 优化与完善
├── 性能优化
├── 监控指标
└── 配置管理

Phase 3（可选）: gRPC 扩展
├── gRPC 客户端实现
├── 服务发现
└── 负载均衡
```

### 架构代码示意

```rust
// src/storage/fulltext/mod.rs

pub enum FulltextBackend {
    /// 内嵌式实现（默认）
    Embedded(TantivyProvider),
    /// gRPC 客户端（可选）
    Grpc(GrpcFulltextClient),
}

impl FulltextProvider for FulltextBackend {
    async fn search(&self, query: &str) -> Result<SearchResults> {
        match self {
            FulltextBackend::Embedded(provider) => provider.search(query).await,
            FulltextBackend::Grpc(client) => client.search(query).await,
        }
    }
    // ... 其他方法
}

/// 根据配置创建后端
pub fn create_backend(config: &FulltextConfig) -> Result<FulltextBackend> {
    match config.backend {
        BackendType::Embedded => {
            let provider = TantivyProvider::new(&config.index_path)?;
            Ok(FulltextBackend::Embedded(provider))
        }
        BackendType::Grpc => {
            let client = GrpcFulltextClient::connect(&config.grpc_endpoint)?;
            Ok(FulltextBackend::Grpc(client))
        }
    }
}
```

## 结论

| 维度 | 内嵌式 | gRPC | 推荐 |
|------|--------|------|------|
| 性能 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | 内嵌式 |
| 运维 | ⭐⭐⭐⭐⭐ | ⭐⭐ | 内嵌式 |
| 扩展性 | ⭐⭐ | ⭐⭐⭐⭐⭐ | gRPC |
| 一致性 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | 内嵌式 |
| 灵活性 | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | gRPC |

**最终建议**：
- **当前阶段**：采用内嵌式架构，快速交付核心功能
- **未来演进**：保留 gRPC 扩展接口，根据需求后续实现
- **混合策略**：通过配置支持两种模式，默认内嵌式
