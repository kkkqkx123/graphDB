# Inversearch 存储系统改进方案

基于 `存储分析.md` 的评估和实际代码审查，结合当前嵌入式库设计模式，以下是调整后的改进方案。

## 设计原则

inversearch-service 是一个**嵌入式库**，通过 adapter 模式被 graphDB 消费（见 `src/search/adapters/inversearch_adapter.rs`）。因此：

1. **Metrics 仅提供编程 API**：库内部只暴露简单的数据快照结构体，不集成任何外部监控系统（Prometheus、JSON 端点等）。外部集成由 adapter 层或宿主应用负责。
2. **配置自包含**：库的配置类型独立定义，不依赖外部配置系统。
3. **StorageInterface 是内部抽象**：存储层是库的内部实现细节，不暴露给库的消费者（`EmbeddedIndex` 用户无需感知存储层）。

## 1. StorageFactory 配置驱动

**问题**: `StorageFactory::from_config` 完全忽略 `Config` 参数，始终返回 `MemoryStorage`。配置文件中的 `StorageConfig.backend` 字段（支持 `ColdWarmCache`）毫无作用。

**改进方案**:

- 读取 `config.storage.backend` 字段，根据配置选择存储后端
- 当 `config.storage.enabled == true` 且 backend 为 `ColdWarmCache` 时，创建 `ColdWarmCacheManager` 实例
- 为 `ColdWarmCacheManager` 添加 `with_config` 方法，从 `Config` 中的 storage 配置读取路径参数

```rust
impl StorageFactory {
    pub async fn from_config(config: &Config) -> Result<Arc<dyn StorageInterface>> {
        if config.storage.enabled {
            match config.storage.backend {
                StorageBackend::ColdWarmCache => {
                    let cache_config = ColdWarmCacheConfig {
                        cold_storage_path: config.storage.base_path
                            .clone()
                            .unwrap_or_else(|| PathBuf::from("./data/cold")),
                        wal_path: config.storage.wal_dir
                            .clone()
                            .unwrap_or_else(|| PathBuf::from("./data/wal")),
                        ..ColdWarmCacheConfig::default()
                    };
                    let manager = ColdWarmCacheManager::with_config(cache_config).await?;
                    Ok(Arc::new(manager))
                }
            }
        } else {
            let storage = MemoryStorage::new();
            Ok(Arc::new(storage))
        }
    }
}
```

**相关文件**:

- `src/storage/factory.rs`
- `src/config/mod.rs` - `StorageConfig` 需扩展 `base_path`、`wal_dir` 等字段

## 2. ColdWarmCacheManager 的 StorageInterface 实现

**问题**: `ColdWarmCacheManager` 实现了 `StorageInterface`，但核心方法全是空实现：

- `commit`: 空操作，不接收任何索引数据
- `get`: 始终返回空结果
- `enrich`: 始终返回空结果
- `has`: 始终返回 false
- `remove`: 空操作

这导致 `ColdWarmCacheManager` 无法作为 `StorageInterface` 的替代品使用。

### 2a. commit 实现

将 `Index` 中的数据提取为 `IndexData`，根据大小决定存入热缓存或直接落盘：

```rust
async fn commit(&self, index: &Index, replace: bool, append: bool) -> Result<()> {
    let data = IndexData::from_index(index);
    let index_name = index.name.as_deref().unwrap_or("default");

    if replace {
        self.hot_cache.remove(index_name);
        self.warm_cache.remove(index_name);
        self.cold_storage.remove(index_name);
    }

    if append {
        if let Some(existing) = self.get_index(index_name).await? {
            let merged = existing.merge(data);
            self.insert_index(index_name, merged).await
        } else {
            self.insert_index(index_name, data).await
        }
    } else {
        self.insert_index(index_name, data).await
    }
}
```

`IndexData` 需要新增 `from_index` 和 `merge` 方法。

### 2b. get 实现

```rust
async fn get(&self, key: &str, ctx: Option<&str>, limit: usize, offset: usize,
             resolve: bool, enrich: bool) -> Result<SearchResults> {
    let mut all_results = Vec::new();
    for entry in self.hot_cache.iter() {
        let data = &entry.data;
        let ids = if let Some(ctx_key) = ctx {
            data.context_data.get(ctx_key)
                .and_then(|m| m.get(key))
                .cloned()
                .unwrap_or_default()
        } else {
            data.data.get(key).cloned().unwrap_or_default()
        };
        all_results.extend(ids);
    }
    // 同样需要检查 warm cache 和 cold storage
    Ok(apply_limit_offset(&all_results, limit, offset))
}
```

### 2c. 设计权衡

当前 `StorageInterface` 的 `get` / `has` / `remove` 是文档级别的操作，而 `ColdWarmCacheManager` 是索引级别的存储。两种方案：

- **方案 A（推荐）**: `ColdWarmCacheManager` 在 commit 时将索引数据展开为文档级索引，存入三层缓存。这样 `get` 可以直接从缓存查询。
- **方案 B**: 保持索引级存储，在 `get` 时遍历所有缓存索引合并结果。性能较差，不推荐。

推荐方案 A，因为 `ColdWarmCacheManager` 的价值在于缓存大数据块，展开为文档级数据会失去缓存优势。但作为嵌入式库的内部实现，这个权衡是合理的——库消费者通过 `EmbeddedIndex` 操作，不直接感知存储粒度。

**相关文件**:

- `src/storage/cold_warm_cache/manager.rs`
- `src/storage/common/trait.rs`

## 3. StorageManager 泛型化

**问题**: `StorageManager` 和 `DefaultStorage` 硬编码为 `MemoryStorage`，无法使用其他存储后端。

```rust
// 当前代码
pub type DefaultStorage = MemoryStorage;

pub struct StorageManager {
    storage: Arc<DefaultStorage>,  // 固定为 MemoryStorage
}
```

**改进方案**: 将 `StorageManager` 改为使用 `Arc<dyn StorageInterface>`：

```rust
pub struct StorageManager {
    storage: Arc<dyn StorageInterface>,
}

impl StorageManager {
    pub fn new(storage: Arc<dyn StorageInterface>) -> Self {
        Self { storage }
    }

    pub async fn open(&self) -> Result<()> {
        self.storage.open().await
    }
    // ... 其他方法委托给 storage
}
```

**注意**: `StorageManager` 是库内部组件，不暴露给 `EmbeddedIndex` 用户。此改动不影响公共 API。

**相关文件**:

- `src/storage/manager.rs`

## 4. WAL 恢复实现 bug

**问题**: `WALManager::read_wal_file` 使用 `load_from_file`（反序列化为 `FileStorageData`），再转为字节。但 WAL 文件是使用 `postcard` 序列化的、长度前缀的 `WALEntry` 条目格式，不兼容 `FileStorageData` 结构。

```rust
// 当前有问题的代码
async fn read_wal_file(&self, path: &PathBuf) -> Result<Vec<WALEntry>> {
    let file_data = load_from_file(path).await?;   // 试图作为 FileStorageData 读取
    let bytes = to_allocvec(&file_data)?;           // 再序列化为字节
    // 然后解析长度前缀的 WALEntry...
}
```

**改进方案**: 直接读取原始字节：

```rust
async fn read_wal_file(&self, path: &PathBuf) -> Result<Vec<WALEntry>> {
    let bytes = tokio::fs::read(path).await?;
    let mut entries = Vec::new();
    let mut offset = 0;

    while offset + 4 <= bytes.len() {
        let len = u32::from_le_bytes(
            bytes[offset..offset + 4].try_into().unwrap()
        ) as usize;
        offset += 4;
        if offset + len > bytes.len() { break; }
        let entry = postcard::from_bytes::<WALEntry>(&bytes[offset..offset + len])
            .map_err(|e| StorageError::Deserialization(e.to_string()))?;
        entries.push(entry);
        offset += len;
    }
    Ok(entries)
}
```

**相关文件**:

- `src/storage/cold_warm_cache/manager.rs` - `WALManager::read_wal_file` 方法

## 5. 温缓存内存映射优化

**问题**: `WarmCacheEntry::load` 完全加载文件到内存，没有利用内存映射。温缓存的设计目标是"内存映射文件"，但实际实现只是普通文件读写。

**改进方案**: 使用 `memmap2` crate 实现内存映射（`memmap2` 已在主项目 `Cargo.toml` 中）：

```rust
use memmap2::Mmap;

impl WarmCacheEntry {
    async fn load(&self) -> Result<FileStorageData> {
        let file = tokio::fs::File::open(&self.file_path).await?;
        let mmap = unsafe { Mmap::map(&file.into_std().await)? };
        let data = if self.compressed {
            let decompressed = decompress_data(&mmap)?;
            postcard::from_bytes(&decompressed)
                .map_err(|e| StorageError::Deserialization(e.to_string()))?
        } else {
            postcard::from_bytes(&mmap)
                .map_err(|e| StorageError::Deserialization(e.to_string()))?
        };
        Ok(data)
    }
}
```

**相关文件**:

- `src/storage/cold_warm_cache/manager.rs`
- `crates/inversearch/Cargo.toml` - 添加 `memmap2` 依赖

## 6. SearchCache 写锁竞争

**问题**: `SearchCache::get_async` 获取写锁，导致读写互斥。LRU 缓存的 `get` 操作需要修改访问顺序（将访问项移到尾部），因此需要写锁。

**当前方案是可接受的权衡**。在嵌入式库场景下，`SearchCache` 是 `Index` 的内部组件，并发压力通常不大。如果未来需要优化：

**优化建议**: 使用 `tokio::sync::RwLock` 配合分片策略，减少锁粒度：

```rust
const SHARD_COUNT: usize = 16;

pub struct SearchCache {
    stores: [Arc<RwLock<LruCache<String, CacheEntry>>>; SHARD_COUNT],
    // ...
}

impl SearchCache {
    fn shard(key: &str) -> usize {
        key.len().wrapping_mul(0x517cc1b727220a95) % SHARD_COUNT
    }
}
```

**相关文件**:

- `src/search/cache.rs`

## 7. 纯磁盘存储实现

**问题**: 当前没有纯磁盘存储实现（不经过缓存层），无法满足只读场景或大数据集需求。

**改进方案**: 新增 `FileStorage` 实现，作为 `StorageInterface` 的一个变体：

```rust
pub struct FileStorage {
    base_path: PathBuf,
    base: RwLock<StorageBase>,
}

impl FileStorage {
    pub fn new(path: PathBuf) -> Self { /* ... */ }

    async fn load_snapshot(&self) -> Result<()> { /* ... */ }
}

#[async_trait]
impl StorageInterface for FileStorage {
    async fn commit(&self, index: &Index, replace: bool, append: bool) -> Result<()> {
        // 更新内存并触发异步刷盘
    }
    // ...
}
```

**注意**: `FileStorage` 是库内部实现，不直接暴露给 `EmbeddedIndex` 用户。用户通过 `StorageFactory::from_config` 选择后端。

**新增文件**:

## 8. 缓存预热机制

**问题**: 系统启动时缓存为空，需要反复查询才能填充热点数据。

**改进方案**: 在 `ColdWarmCacheManager::recover` 中增加预热逻辑。预热列表由库内部管理（例如记录上次关闭时的热缓存索引名），不依赖外部配置文件：

```rust
impl ColdWarmCacheManager {
    /// 从 checkpoint 或上次关闭时的热缓存快照中恢复热点数据
    async fn warmup_from_checkpoint(&self) -> Result<()> {
        if let Some(warmup_list) = self.load_warmup_list().await? {
            for index_name in warmup_list {
                if let Some(data) = self.load_from_cold_storage(&index_name).await? {
                    self.promote_to_hot(&index_name, data).await?;
                }
            }
        }
        Ok(())
    }

    /// 在创建 checkpoint 时记录当前热缓存索引名
    async fn save_warmup_list(&self) -> Result<()> {
        let hot_keys: Vec<String> = self.hot_cache.iter().map(|e| e.key().clone()).collect();
        // 序列化到 checkpoint 元数据中
    }
}
```

## 9. 配置系统完善

**问题**: `ColdWarmCacheConfig` 硬编码默认值，不通过 `Config` 结构体配置。

**改进方案**: 扩展 `StorageConfig` 以包含冷热缓存参数。注意这些配置类型是库自身的，不依赖外部系统：

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct StorageConfig {
    pub enabled: bool,
    pub backend: StorageBackend,
    pub base_path: Option<PathBuf>,
    pub wal_dir: Option<PathBuf>,

    // 冷热缓存参数
    pub hot_cache_max_size_mb: Option<usize>,
    pub warm_cache_max_size_mb: Option<usize>,
    pub wal_enabled: Option<bool>,
    pub wal_max_files: Option<usize>,
}
```

## 10. 指标监控（仅编程 API）

**问题**: `CacheStats` 当前只提供 `get_stats()` 返回格式化字符串，不利于宿主应用采集。

**改进方案**: 提供纯数据快照结构体，不引入任何外部监控依赖。宿主应用（如 graphDB 的 adapter 层）可自行决定如何消费这些数据（Prometheus、JSON 等）：

```rust
/// 纯数据快照，无外部依赖
#[derive(Debug, Clone)]
pub struct CacheStatsSnapshot {
    pub hot_hit: u64,
    pub warm_hit: u64,
    pub cold_hit: u64,
    pub miss: u64,
    pub hot_cache_size_bytes: u64,
    pub warm_cache_size_bytes: u64,
    pub hit_rate: f64,
}

impl CacheStats {
    /// 返回当前统计信息的快照
    pub fn snapshot(&self) -> CacheStatsSnapshot {
        CacheStatsSnapshot {
            hot_hit: self.hot_hit.load(Ordering::Relaxed),
            warm_hit: self.warm_hit.load(Ordering::Relaxed),
            cold_hit: self.cold_hit.load(Ordering::Relaxed),
            miss: self.miss.load(Ordering::Relaxed),
            hot_cache_size_bytes: self.hot_cache_size.load(Ordering::Relaxed),
            warm_cache_size_bytes: self.warm_cache_size.load(Ordering::Relaxed),
            hit_rate: self.hit_rate(),
        }
    }
}
```

**外部集成说明**（非本库职责）：

- Prometheus 格式导出 → 在 graphDB 的 adapter 层或监控中间件中实现
- JSON HTTP 端点 → 在 graphDB 的 Web 服务层实现
- 日志输出 → 在宿主应用中配置

**相关文件**:

- `src/storage/cold_warm_cache/manager.rs`
