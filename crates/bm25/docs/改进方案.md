# BM25 包改进方案

## 问题分析

根据 `存储分析.md` 的分析结果，结合代码审查和当前嵌入式库设计模式，`crates/bm25` 包存在以下关键问题。

### 设计原则

bm25-service 是一个**嵌入式库**，通过 adapter 模式被 graphDB 消费（见 `src/search/adapters/bm25_adapter.rs`）。因此：

1. **存储层是内部实现细节**：`TantivyStorage`、`StorageInterface` 等是库内部组件，库消费者通过 `Bm25Index`（embedded API）操作，不直接感知存储层。
2. **Metrics 仅提供编程 API**：库内部只暴露简单的数据快照结构体，不集成任何外部监控系统。外部集成由 adapter 层或宿主应用负责。
3. **配置自包含**：库的配置类型独立定义，不依赖外部配置系统。

### 当前问题清单

#### 1. BM25 统计信息未持久化

`TantivyStorage` 中的 `commit_stats()` 和 `commit_batch()` 为空实现，导致 `stats_extractor.rs` 提取的 TF/DF 统计信息被丢弃。详细影响：

- `api/core/document.rs` 中的 `add_document_with_storage()` 调用 `storage.commit_batch()` 提交统计信息，但无实际效果
- `api/core/batch.rs` 中的 `batch_add_documents_with_storage()` 同理
- `api/core/delete.rs` 中的 `delete_document_with_storage()` 调用 `storage.delete_doc_stats()`，同样为空实现
- 结果是查询时 `get_stats()` 只能从 Tantivy 倒排索引重新计算 df，无法获取已持久化的精确统计数据

#### 2. `get_tf()` 实现错误

`get_tf()` 忽略 `doc_id` 参数，返回全局比例 `doc_freq / total_docs`，而非真正针对特定文档的词项频率。这在 BM25 算法中会导致评分不准确。

#### 3. 统计信息不完整

- `get_stats()` 返回的 `tf` 字段始终为空 `HashMap`
- `avg_doc_length` 通过 `num_docs * 100 / num_docs` 估算，结果始终为 100.0
- 多个字段（content、title）的统计信息未区分

#### 4. 两层索引管理冗余

系统中存在两层独立的索引管理：

- `storage/tantivy.rs` 中的 `TantivyStorage`——管理自己的 Index、IndexWriter、IndexReader
- `api/core/index.rs` 中的 `IndexManager`——也管理独立的 Index 和 IndexReader

两者各自管理同一个磁盘路径的 Tantivy 索引，但没有共享机制，造成资源浪费和潜在的锁冲突。

## 架构调整

### 当前架构

```
+------------------+     +------------------+
|   API Core       | --> |  Storage Layer   |
| (IndexManager)   |     | (TantivyStorage) |
+------------------+     +------------------+
                                |
                                v
                         +------------------+
                         |  Tantivy Index   |
                         |  (on disk)       |
                         +------------------+
```

### 调整后架构

```
+------------------+     +------------------+
|   API Core       | --> |  Storage Layer   |
| (IndexManager)   |     | (TantivyStorage) |
+------------------+     +------------------+
         |                       |
         |                       v
         |                +------------------+
         |                |  Tantivy Index   |
         |                |  (on disk)       |
         |                +------------------+
         |
         v
+---------------------------+
|  StatsStore (独立文件存储)  |
|  (postcard 序列化到磁盘)    |
+---------------------------+
```

**关键变化**：

- 统计信息不写入 Tantivy 索引（避免引入第二个 Tantivy schema 的复杂度）
- 使用独立的 `StatsStore`，以 postcard 格式序列化到单独的文件
- `StatsStore` 是 `TantivyStorage` 的内部组件，不暴露给外部

---

## 具体改进项

### 改进一：统计信息持久化（P0）

**目标**：让 `commit_stats()` 和 `commit_batch()` 真正将 TF/DF 写入持久化存储。

**方案**：使用独立的 `StatsStore`，以 postcard 格式将统计信息序列化到单独的文件。不引入第二个 Tantivy 索引，避免增加系统复杂度。

#### StatsStore 实现

```rust
use crate::error::Result;
use crate::storage::common::Bm25Stats;
use postcard::{from_bytes, to_allocvec};
use std::collections::HashMap;
use std::path::PathBuf;
use tokio::sync::RwLock;

/// 统计信息存储
///
/// 将 BM25 统计信息以 postcard 格式持久化到独立文件。
/// 不依赖 Tantivy 索引，避免引入第二个 Schema 的复杂度。
pub struct StatsStore {
    /// 统计信息文件路径
    path: PathBuf,
    /// 内存中的统计信息缓存
    stats: RwLock<Bm25Stats>,
}

impl StatsStore {
    pub fn new(path: PathBuf) -> Self {
        Self {
            path,
            stats: RwLock::new(Bm25Stats::default()),
        }
    }

    /// 从磁盘加载统计信息
    pub async fn load(&self) -> Result<Bm25Stats> {
        if !self.path.exists() {
            return Ok(Bm25Stats::default());
        }
        let bytes = tokio::fs::read(&self.path).await?;
        let stats = from_bytes::<Bm25Stats>(&bytes)
            .map_err(|e| Bm25Error::Deserialization(e.to_string()))?;
        *self.stats.write().await = stats.clone();
        Ok(stats)
    }

    /// 批量提交统计信息
    pub async fn commit_batch(&self, new_stats: &Bm25Stats) -> Result<()> {
        let mut stats = self.stats.write().await;
        // 合并统计信息
        for (term, df) in &new_stats.df {
            stats.df.insert(term.clone(), *df);
        }
        for (term, tf) in &new_stats.tf {
            *stats.tf.entry(term.clone()).or_insert(0.0) += tf;
        }
        stats.total_docs = new_stats.total_docs;
        stats.avg_doc_length = new_stats.avg_doc_length;
        // 持久化到磁盘
        self.flush(&stats).await
    }

    /// 获取指定词项的统计信息
    pub async fn get_stats(&self, term: &str) -> Result<Option<Bm25Stats>> {
        let stats = self.stats.read().await;
        let df = stats.df.get(term).copied();
        let tf = stats.tf.get(term).copied();
        match (df, tf) {
            (Some(df), _) => Ok(Some(Bm25Stats {
                tf: tf.map(|v| HashMap::from([(term.to_string(), v)])).unwrap_or_default(),
                df: HashMap::from([(term.to_string(), df)]),
                total_docs: stats.total_docs,
                avg_doc_length: stats.avg_doc_length,
            })),
            _ => Ok(None),
        }
    }

    /// 获取文档频率
    pub async fn get_df(&self, term: &str) -> Result<Option<u64>> {
        let stats = self.stats.read().await;
        Ok(stats.df.get(term).copied())
    }

    /// 删除文档的统计信息
    pub async fn delete_doc_stats(&self, doc_id: &str) -> Result<()> {
        let mut stats = self.stats.write().await;
        // 从 tf 中移除该文档的词项频率
        stats.tf.retain(|_, _| true); // 简化处理：实际需要按 doc_id 过滤
        self.flush(&stats).await
    }

    /// 刷新到磁盘
    async fn flush(&self, stats: &Bm25Stats) -> Result<()> {
        let bytes = to_allocvec(stats)
            .map_err(|e| Bm25Error::Serialization(e.to_string()))?;
        tokio::fs::write(&self.path, &bytes).await?;
        Ok(())
    }
}
```

#### get_df() 改进

优先从 `StatsStore` 查询，fallback 到 Tantivy 索引：

```rust
async fn get_df(&self, term: &str) -> Result<Option<u64>> {
    // 优先从 StatsStore 查询
    if let Some(ref store) = self.stats_store {
        if let Some(df) = store.get_df(term).await? {
            return Ok(Some(df));
        }
    }
    // fallback: 从 Tantivy 主索引查询
    let reader = self.reader.as_ref()
        .ok_or_else(|| Bm25Error::IndexNotInitialized)?;
    let reader = reader.read().await;
    let searcher = reader.searcher();
    let field = self.schema.get_field("content").unwrap();
    let term_obj = Term::from_field_text(field, term);
    let doc_freq = searcher.doc_freq(&term_obj)?;
    Ok(Some(doc_freq as u64))
}
```

### 改进二：修复 get_tf() 实现（P1）

**目标**：返回真实的每文档词项频率，而非全局文档比例。

**方案**：利用 Tantivy 的 `SegmentReader` 的 `term_statistics()` API，直接从倒排索引中获取某词在特定文档中的出现次数，避免重新分词。

```rust
async fn get_tf(&self, term: &str, doc_id: &str) -> Result<Option<f32>> {
    let reader = self.reader.as_ref()
        .ok_or_else(|| Bm25Error::IndexNotInitialized)?;
    let reader = reader.read().await;
    let searcher = reader.searcher();

    // 找到目标文档的地址
    let doc_id_field = self.schema.get_field("document_id")?;
    let term_query = Term::from_field_text(doc_id_field, doc_id);
    let query = TermQuery::new(term_query, IndexRecordOption::Basic);

    let top_docs = searcher.search(&query, &TopDocs::with_limit(1))?;
    let (_score, doc_addr) = match top_docs.into_iter().next() {
        Some(result) => result,
        None => return Ok(Some(0.0)),
    };

    // 利用 segment reader 获取词项在文档中的频率
    let content_field = self.schema.get_field("content")?;
    let term = Term::from_field_text(content_field, term);

    for segment_reader in searcher.segment_readers() {
        if segment_reader.doc_id_range().contains(&doc_addr.into()) {
            let inverted_index = segment_reader.inverted_index(content_field)?;
            let term_info = inverted_index.read_term(&term)?;
            if let Some(info) = term_info {
                // 查找该文档在倒排列表中的位置
                let mut postings = inverted_index.read_postings(&term, IndexRecordOption::Basic)?;
                while postings.advance() {
                    if postings.doc() == doc_addr.into() {
                        return Ok(Some(postings.term_freq() as f32));
                    }
                }
            }
            break;
        }
    }
    Ok(Some(0.0))
}
```

### 改进三：修复 avg_doc_length 计算（P1）

**目标**：基于实际文档计算平均文档长度。

**方案**：在文档写入时计算文档长度（所有词项的 TF 之和），持久化到 `StatsStore`。

#### 写入时计算

在 `add_document_with_storage()` 中，从提取出的 TF 统计信息计算文档长度：

```rust
pub async fn add_document_with_storage(
    manager: &IndexManager,
    storage: &MutableStorageManager,
    schema: &IndexSchema,
    document_id: &str,
    fields: &HashMap<String, String>,
    avg_doc_length: f32,
) -> Result<()> {
    let stats = extract_tf_df_stats(fields, 0, avg_doc_length);
    let doc_length: f32 = stats.tf.values().sum();  // 实际文档长度
    // 更新全局 avg_doc_length
    let total_docs = manager.count()?;
    let new_avg = if total_docs > 0 {
        (avg_doc_length * total_docs as f32 + doc_length) / (total_docs as f32 + 1.0)
    } else {
        doc_length
    };
    let updated_stats = Bm25Stats {
        avg_doc_length: new_avg,
        ..stats
    };
    storage.commit_batch(&updated_stats).await?;
    // ...其余逻辑...
}
```

### 改进五：合并 IndexManager 与 TantivyStorage 中的索引管理（P2）

**目标**：消除两套独立的 Tantivy 索引管理层。

当前系统中：

- `api/core/index.rs` 的 `IndexManager` 负责管理索引、读写器、reader 缓存，与搜索功能紧密耦合
- `storage/tantivy.rs` 的 `TantivyStorage` 也管理独立的一套索引、读写器，但与搜索功能无关，仅用于统计信息

**短期方案**：
保持两层独立，但确保访问同一磁盘路径时采用适当的锁机制协调，避免冲突。`StatsStore` 使用独立文件，不与 Tantivy 索引共享路径，消除冲突风险。

**长期方案**：
将 `TantivyStorage` 重构为仅依赖 `IndexManager` 的瘦封装，复用其 Index/IndexReader，仅在统计信息写入时使用独立的 `StatsStore`：

```rust
pub struct TantivyStorage {
    index_manager: Arc<IndexManager>,
    stats_store: Option<StatsStore>,
}
```

## 风险与注意事项

1. **向后兼容性**：项目当前不要求向后兼容，但 `StatsStore` 使用 postcard 格式，与现有 Tantivy 索引格式不同，需要清理或迁移现有统计信息数据。
2. **并发安全**：`StatsStore` 使用 `tokio::sync::RwLock`，在单节点嵌入式场景下性能足够。
3. **StatsStore 文件大小**：统计信息文件通常很小（数千个词项约几十 KB），不会成为性能瓶颈。
4. **统计信息一致性**：`commit_batch()` 写入后，读取时 `StatsStore` 和 Tantivy 主索引之间存在短暂不一致，但 BM25 评分对微小延迟不敏感。
5. **不引入应用层缓存**：当前方案不添加 LRU 缓存层。`StatsStore` 本身在内存中维护全量统计信息，查询无需磁盘 I/O。如果未来统计信息量极大（百万级词项），可考虑在 adapter 层添加缓存，而非侵入库内部。
